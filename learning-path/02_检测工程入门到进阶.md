# 检测工程入门到进阶

检测工程不是“写一条规则”那么简单，而是围绕数据、行为、噪声和处置做系统化建设。

## 一、先问自己四个问题

1. 我想检测什么行为，而不是哪个关键词？
2. 哪些日志源真的能支撑这个判断？
3. 命中后谁来处理，处理动作是什么？
4. 规则失败时，我如何知道是漏报还是数据缺失？

## 二、检测规则的常见类型

### 1. IOC 型

特点：

- 实现快
- 易理解
- 生命周期短
- 绕过成本低

适合：

- 热点事件快速响应
- 已知恶意样本回溯
- 紧急补洞期临时监测

### 2. 行为型

特点：

- 更稳健
- 依赖上下文
- 对数据质量要求更高

适合：

- 账号异常
- 横向移动
- 爆破与探测
- 异常上传与落地
- 稀有进程链

### 3. 序列关联型

特点：

- 更接近真实攻击链
- 能降低单点误报
- 建设成本更高

例如：

- Web 异常请求后接主机异常进程
- 管理员登录后接权限变更
- 外部下载后接计划任务创建

## 三、检测工程的核心工作流

- 确定用例
- 盘点数据源
- 写初版规则
- 用历史数据回放验证
- 标记误报来源
- 调整阈值与白名单
- 明确处置动作
- 定期复盘规则效果

## 四、容易被忽略的点

- 日志字段不统一导致规则不可迁移
- 告警没有 owner，导致命中也没人处理
- 只有规则没有处置手册
- 没有版本记录，调优过程不可追踪
- 只看单点命中，不看上下文聚合

## 五、建议沉淀的仓库资产

- `detections/`：规则文件
- `playbooks/`：命中后的调查步骤
- `samples/`：测试样本与脱敏日志
- `reports/`：规则效果评估
- `tools/`：数据转换与校验脚本

## 六、对个人成长最有用的练习

- 把一次真实告警复盘写成规则优化记录
- 把热点漏洞转换成行为检测条件
- 用自己的日志样本做回放测试
- 比较 IOC 规则与行为规则的命中差异
