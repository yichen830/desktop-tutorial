# Fastjson 历史反序列化风险防御分析

Fastjson 这类问题不太适合死记某一个编号，更适合当成一类长期风险去看。

因为它本质上是：

- 组件能力太强
- 输入边界没收住
- 老版本长期留在生产里

## 真正要排查的点

重点不是“项目里有没有 Fastjson”这么简单，而是：

- 版本老不老
- 有没有处理不可信 JSON 输入
- autoType 相关风险有没有收住
- 反序列化入口是不是暴露在外网

## 这类问题为什么烦

因为很多项目都把它当基础库：

- 控制器收 JSON
- 内部接口转对象
- 消息队列消费
- 配置解析

所以它的风险点可能散在很多地方。

## 排查时先看哪几类位置

- Web 接口入参处理
- RPC 或内部接口
- MQ 消费逻辑
- 配置加载和工具类
- 通用 JSON 转对象封装

## 修复思路

- 优先升级
- 收紧危险配置
- 对不可信输入做白名单思路处理
- 审查通用工具类，别只看业务代码
- 把历史风险版本做一次统一盘点

## 复盘时值得记一笔

这类问题最容易暴露团队的一个短板：

基础组件版本没人持续治理，最后问题会反复以不同形式出现。
